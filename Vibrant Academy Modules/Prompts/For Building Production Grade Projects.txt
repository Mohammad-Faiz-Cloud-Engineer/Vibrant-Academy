═══════════════════════════════════════════════════════════════════════════════
MANDATORY PRE-CODING PROTOCOL (Execute Before Writing Any Code)
═══════════════════════════════════════════════════════════════════════════════

Before implementing ANY feature, you MUST complete these phases in order:

PHASE 1 — ARCHITECTURE & THREAT ANALYSIS
─────────────────────────────────────────
1. SCOPE DECOMPOSITION
   □ List exact deliverables (no assumptions)
   □ Define input/output contracts for every function/module
   □ Identify all edge cases and failure scenarios
   □ Map external dependencies and their failure modes
   □ State what is OUT OF SCOPE explicitly

2. SECURITY THREAT MODEL
   □ List every attack surface in this feature
   □ Identify which inputs can be malicious
   □ Classify sensitive data and protection mechanisms
   □ Define auth/authz requirements
   □ Map data flow for potential injection points

3. TECHNICAL ARCHITECTURE
   □ Choose design pattern (justify why over alternatives)
   □ Define folder/module structure
   □ State data flow (text diagram acceptable)
   □ Identify optimization opportunities

PHASE 2 — TEST-DRIVEN DEVELOPMENT (NON-NEGOTIABLE)
───────────────────────────────────────────────────
Write ALL tests BEFORE implementation. Tests must include:

UNIT TESTS:
   □ Happy path for every function
   □ Every edge case from Phase 1
   □ Null/undefined/empty input handling
   □ Type boundary conditions (max int, empty string, zero, negative)
   □ Invalid type injection attempts

INTEGRATION TESTS:
   □ Component/module interaction
   □ API contract verification
   □ Database operation correctness (if applicable)
   □ External service failure simulation

SECURITY TESTS:
   □ SQL/NoSQL injection payloads
   □ XSS payload inputs (if web/UI involved)
   □ Auth bypass attempts (if auth involved)
   □ Oversized/malformed input handling
   □ Rate limiting validation

PERFORMANCE TESTS (flag if applicable):
   □ Large dataset behavior (10k+ records)
   □ Concurrent call behavior
   □ Memory leak indicators (long-running scenarios)

Test Naming Convention (STRICT):
   test_[unit]_[scenario]_[expectedResult]
   Example: test_parseUserInput_emptyString_returnsValidationError

PHASE 3 — IMPLEMENTATION UNDER CONSTRAINTS
────────────────────────────────────────
Write code that passes ALL tests under these hard constraints:

CODE QUALITY CONSTRAINTS:
   □ Single responsibility per function (max 40 lines)
   □ Max nesting depth: 3 levels (extract beyond that)
   □ Zero magic numbers/strings → named constants only
   □ Zero commented-out code (remove or explain)
   □ Zero dead code (unused variables, unreachable branches)
   □ Zero debug statements in production paths
   □ Zero emojis in code/comments
   □ Naming follows language conventions (camelCase JS/TS, snake_case Python)

SECURITY CONSTRAINTS (ZERO TOLERANCE):
   □ NO hardcoded secrets, tokens, API keys, passwords → env vars only
   □ ALL user/external inputs validated AND sanitized before use
   □ Parameterized queries ONLY → zero string-concatenated SQL
   □ NO sensitive data (tokens, PII, passwords) in logs
   □ HTTPS enforced for all external calls
   □ Error messages NEVER expose internal stack traces to users
   □ Auth tokens: validate algorithm, expiry, signature
   □ Passwords: never stored/transmitted plain text
   □ Sessions: proper invalidation on logout
   □ CORS: restrictive, not permissive
   □ CSRF protection on state-changing endpoints
   □ Cookies: httpOnly, secure, sameSite flags

OPTIMIZATION CONSTRAINTS:
   □ No synchronous blocking where async is possible
   □ No N+1 query patterns
   □ Cleanup: remove event listeners, clear timers, unsubscribe
   □ Use memoization where result reuse is valid
   □ Lazy-load where applicable (heavy imports, images, routes)
   □ API calls must have: error handling + loading state + timeout

ERROR HANDLING CONSTRAINTS:
   □ Every async operation wrapped in try/catch or .catch()
   □ Every error caught AND handled (no silent swallowing)
   □ Meaningful error messages for debugging (server-side only)
   □ Graceful degradation for non-critical failures

TYPE SAFETY (if TypeScript):
   □ No `any` type without explicit justification comment
   □ All function parameters and return types typed
   □ Strict mode enabled

═══════════════════════════════════════════════════════════════════════════════
AUTONOMOUS CODEBASE AUDIT PROTOCOL (Execute on Existing Code)
═══════════════════════════════════════════════════════════════════════════════

When auditing existing code, systematically check EVERY file against:

BUGS & LOGIC ERRORS:
   □ Wrong conditionals, inverted logic, off-by-one errors
   □ Incorrect/missing return values
   □ Wrong variable scope or reference
   □ Async/await misuse, unhandled promises, race conditions
   □ Infinite loops, missing break conditions
   □ State bugs, stale closures, wrong hook dependencies (React)

SECURITY AUDIT:
   □ Hardcoded secrets anywhere in code
   □ SQL/NoSQL injection (string concatenation in queries)
   □ Missing input validation on every entry point
   □ XSS via dangerouslySetInnerHTML, innerHTML, eval()
   □ Routes missing auth/authorization middleware
   □ Insecure direct object references (IDOR)
   □ JWT validation flaws (algorithm, expiry, signature)
   □ Passwords in plain text (storage or logs)
   □ Sessions not invalidated on logout
   □ CORS misconfiguration
   □ Missing CSRF protection
   □ Cookies missing security flags
   □ Sensitive data in API responses
   □ Stack traces exposed to client

PERFORMANCE AUDIT:
   □ N+1 database queries
   □ Missing indexes on filtered/sorted/joined columns
   □ Redundant loops, unnecessary recomputation
   □ Heavy sync operations blocking event loop
   □ Memory leaks, unclosed connections
   □ Missing caching opportunities
   □ Unnecessary re-renders (React: missing useMemo/useCallback)

CODE QUALITY AUDIT:
   □ Dead code, unused variables/imports/functions → DELETE
   □ Duplicated logic → abstract into shared functions
   □ Magic numbers/strings → named constants
   □ Functions doing too many things → split them
   □ Deeply nested if/else → flatten or extract
   □ Misleading/vague names → rename
   □ Inconsistent patterns → normalize
   □ Missing TypeScript types or wrong annotations
   □ Debug statements in production code → remove
   □ Emojis in code → remove all

ERROR HANDLING AUDIT:
   □ Missing try/catch on I/O, network, DB, async operations
   □ Silent catch blocks (do nothing) → fix them
   □ Generic errors with no useful message
   □ Unhandled promise rejections
   □ No fallback for failed external calls

API LAYER AUDIT:
   □ Inconsistent HTTP status codes
   □ Missing request validation (body, params, query)
   □ No pagination on list endpoints
   □ Inconsistent response structure

DATABASE AUDIT:
   □ Queries without parameterization
   □ Missing transactions where multiple writes must be atomic
   □ Schema and model mismatch

CONFIG & ENVIRONMENT AUDIT:
   □ No validation of required env variables on startup
   □ Dev config leaking into production
   □ Secrets in code instead of environment variables

ARCHITECTURE AUDIT:
   □ Folder structure unclear/illogical
   □ Poor separation of concerns
   □ Circular dependencies
   □ Configuration scattered instead of centralized
   □ Duplicate constants across files
   □ Copy-pasted utility functions
