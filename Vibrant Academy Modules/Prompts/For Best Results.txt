TEST COVERAGE MATRIX — Check every box:

UNIT TESTS
☐ Happy path: normal input produces expected output
☐ Zero/empty/null cases for every parameter
☐ Boundary values: MIN, MAX, just-above, just-below
☐ Type coercion failures: string where number expected, etc.
☐ Invalid format: malformed JSON, wrong encoding, special chars

INTEGRATION TESTS
☐ Module A calls Module B correctly
☐ Database operations (if any): CRUD + transaction rollback
☐ External API calls: success + failure + timeout paths
☐ File system operations: read/write/permission errors

SECURITY TESTS
☐ Injection attempts (SQL, NoSQL, Command, LDAP, XPath)
☐ XSS payloads in all text inputs (<script>, javascript:, onerror=)
☐ Path traversal (../../../etc/passwd)
☐ Oversized inputs: 10MB string, 1M array elements, deep nesting
☐ Auth bypass: missing token, expired token, malformed token

CONCURRENCY TESTS (if applicable)
☐ Race condition: two simultaneous writes to same resource
☐ Deadlock potential: circular resource acquisition
☐ Resource exhaustion: connection pool depletion

TEST NAMING STANDARD:
  test_[unit]_[condition]_[expected]
  Examples:
  • test_calculateDiscount_negativePrice_throwsValidationError
  • test_parseCSV_emptyFile_returnsEmptyArray
  • test_authMiddleware_expiredToken_returns401

FRAMEWORK: Use real test framework (Jest, Vitest, Pytest, Go testing, JUnit, etc.)
NO: Placeholder tests, commented "TODO: test", pseudo-code tests
