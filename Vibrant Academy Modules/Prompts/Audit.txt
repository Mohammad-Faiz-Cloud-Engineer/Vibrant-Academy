═══════════════════════════════════════════════════════════════════════════════
PHASE 1: REPOSITORY HYGIENE & CLEANUP (Execute First)
═══════════════════════════════════════════════════════════════════════════════

STEP 1.1 — FILE SYSTEM SCAN
Scan every file and folder. Flag for IMMEDIATE REMOVAL:

[CRITICAL — SECURITY RISK]
□ Secrets & Credentials: .env, .env.*, *.pem, *.key, *.cert, *.p12, 
  config/secrets.*, serviceAccountKey.json, firebase-adminsdk*.json, 
  credentials.json, any file containing API keys, passwords, tokens
□ Backup Files: *.bak, *.orig, *_backup.*, *_old.*, *_copy.*
□ Personal Dev Notes: TODO.txt, NOTES.txt, scratch.*, test123.*, personal logs

[HIGH — REPOSITORY BLOAT]
□ Dependencies: node_modules/, vendor/, __pycache__/, *.pyc, *.pyo, 
  *.egg-info/, .eggs/, target/, bin/, obj/
□ Build Output: dist/, build/, .next/, out/, .nuxt/, .cache/, coverage/
□ Test Artifacts: .nyc_output/, htmlcov/, .pytest_cache/, *.lcov, 
  test-results/, junit.xml

[MEDIUM — ENVIRONMENT NOISE]
□ OS Files: .DS_Store, Thumbs.db, desktop.ini, *.log, *.tmp, *.temp, 
  *.cache, npm-debug.log*, yarn-debug.log*, yarn-error.log*, *.pid, 
  *.seed, *.pid.lock
□ IDE Files: .idea/, *.suo, *.ntvs*, *.njsproj, *.sln, *.sw?, *.iml, 
  .project, .classpath, .vscode/settings.json (unless shared config)
□ Misc: Empty folders, duplicate files, commented-out files

STEP 1.2 — .GITIGNORE AUDIT & ENFORCEMENT
□ Verify .gitignore exists at project root
□ If missing: CREATE comprehensive .gitignore covering all categories above
□ Verify no tracked files are slipping through (git check-ignore -v)
□ Ensure .gitignore itself is not ignored

═══════════════════════════════════════════════════════════════════════════════
PHASE 2: STATIC CODE ANALYSIS (Line-by-Line Review)
═══════════════════════════════════════════════════════════════════════════════

For EVERY source file, execute these checks in order:

2.1 DEAD CODE ELIMINATION
□ Remove all commented-out code blocks (// TODO: remove this, /* old code */)
□ Remove unused variables (declared but never referenced)
□ Remove unused imports/requires/using statements
□ Remove unused functions, methods, components, classes, modules
□ Remove unreachable code (after return statements, inside impossible conditions)
□ Consolidate duplicate logic into shared utilities
□ Remove all emojis from code and comments

2.2 CODE QUALITY VERIFICATION
□ Single Responsibility Principle: Each function does ONE thing only
□ Function length: FLAG if >50 lines, REQUIRE refactoring if >80 lines
□ Nesting depth: FLAG if >3 levels, REQUIRE extraction if >4 levels
□ Magic numbers/strings: Move to named constants or config files
□ Naming conventions: Enforce per-language standards (camelCase JS/TS, 
  snake_case Python, PascalCase classes)
□ Debug statements: REMOVE all console.log, print(), debugger, dd(), 
  var_dump(), console.error/warn (unless intentional error reporting)
□ TODO/FIXME comments: Resolve or convert to tracked issues (not code comments)
□ TypeScript: FLAG any `any` types without explicit explanatory comments

2.3 LOGIC VERIFICATION (Manual Trace)
□ Trace major functions: Verify correctness and logical completeness
□ Conditionals: Check for missing edge cases and boundary conditions
□ Loops: Verify termination conditions, infinite loop risks, off-by-one errors
□ Error handling: Comprehensive try/catch, .catch() chains, error boundaries
□ Async/await: No unhandled promises, race conditions, or callback hell
□ API calls: Error handling + loading states + timeout management
□ Input validation: ALL user inputs, query params, external data validated

2.4 SECURITY AUDIT (Zero Tolerance)
□ Secrets: NO hardcoded tokens, API keys, passwords in source files
□ SQL/NoSQL: NO string concatenation in queries (parameterized only)
□ Injection: User input sanitized and validated before use
□ Logging: NO sensitive data (PII, tokens, credentials) logged
□ Dependencies: Check for critically outdated or known-vulnerable packages
□ Headers: Verify CORS, CSP, HSTS configuration (if applicable)
□ Transport: HTTPS/TLS enforced for all sensitive operations
□ Auth: Verify authentication at all protected endpoints
□ Authorization: Verify permission checks at sensitive operations

═══════════════════════════════════════════════════════════════════════════════
PHASE 3: ARCHITECTURE & PERFORMANCE REVIEW
═══════════════════════════════════════════════════════════════════════════════

3.1 STRUCTURAL INTEGRITY
□ Folder structure: Clean, logical, immediately understandable
□ File placement: Matches purpose and responsibility
□ Separation of concerns: UI, business logic, data access, utilities, config
□ Dependencies: No circular dependencies detected
□ Configuration: Centralized in config files, not scattered
□ Constants: Defined once in shared location, not duplicated
□ Utilities: Extracted and shared, not copy-pasted

3.2 PERFORMANCE OPTIMIZATION
□ React/Vue/Angular: Check for unnecessary re-renders (memoization, 
  proper dependency arrays, useMemo/useCallback where beneficial)
□ Database: No N+1 query patterns, inefficient calls flagged
□ Async: No synchronous blocking operations on event loop
□ Imports: Tree-shaking optimized, no unused heavy imports
□ Memory: Event listeners cleaned up, subscriptions unsubscribed, 
  timers cleared (check componentWillUnmount, useEffect cleanup, etc.)
□ Caching: Opportunities identified (HTTP caching, memoization, 
  client-side storage where appropriate)

═══════════════════════════════════════════════════════════════════════════════
PHASE 4: DOCUMENTATION & TESTING VALIDATION
═══════════════════════════════════════════════════════════════════════════════

4.1 DOCUMENTATION CHECK
□ README.md exists with:
  - Clear project description
  - Installation and run instructions
  - Required environment variables (keys only, no values)
  - Build and deployment process
  - Contribution guidelines
□ Complex functions: Explanatory comments (intent, not mechanics)
□ Cryptic code: Clarifying comments where logic is non-obvious

4.2 TESTING & QUALITY GATES
□ Test coverage exists for critical functions, edge cases, error paths
□ Framework: Production-grade (Jest, Pytest, Vitest, JUnit, etc.)
□ Test quality: Isolated, independent of execution order
□ Mocking: External dependencies properly mocked/stubbed
□ Naming: Clear, descriptive test names
□ Run linters: ESLint, Prettier, Black, etc. — zero violations
□ Run type checkers: TypeScript, mypy, etc. — zero errors
□ Metrics: Report test coverage %, cyclomatic complexity hotspots, 
  duplication ratios

NON-NEGOTIABLE RULES
